    前后端接口规范

    # 简介

    以前的JavaWeb项目大多数都是java程序员又当爹又当妈，又搞前端，又搞后端。随着时代的发展，渐渐的许多大中小公司开始把前后端的界限分的越来越明确， 前端工程师只管前端的事情，后端工程师只管后端的事情。正所谓术业有专攻，一个人如果什么都会，那么他毕竟什么都不精。

    # 开发原则

    • 前后端接口按照粗浅的REST规则制定
      1. 使用GET、POST、PUT、DELETE共4个HTTP Method，而非简单的GET和POST两者。
      2. 响应使用HTTP状态码来标志请求的执行结果，而非以往的success字段。
      3. URL符合业界普遍接受的REST规则，减少在URL中标识操作类型的情况，如使用POST /users代替POST /users/save。
    • 出于技术的限制，对于一些特殊的场景，接口会在REST设计的基础之上进行一些妥协，具体参考各接口规范说明。
    • 在REST接口规范中，URL符合以下约束：
      1. 实体名称均以复数形式表示，如GET /users或POST /users/status。
      2. URL最后 不得 添加斜杠/符号。
      3. 对于GET请求，数据在URL后通过标准的query格式提供，如GET /users?status=1,2&keyword=hello。
      4. 对于各系统，可以在URL前统一添加固定的前缀，如GET /api/{entities}/{id}，前缀应尽量固定。可以根据请求的类型（如是否需要用户登录授权等）使用不同的前缀。规范推荐按以下规则使用前缀： 如果是需要登录授权的接口，以/api/js为前缀。 其它接口，通常没有权限控制，以/api/tool为前缀。

    # 数据类型

    • 统一返回JSON数据类型

    # 接口文档

    接口文档应当 至少 包含以下内容：

    1. 接口的名称及简单的作用描述。
    2. 接口的URL和请求时使用的HTTP Method。
    3. 接口的请求格式及参数。
    4. 接口可能返回的状态码，及每个状态码下的响应类型和格式。
    5. 接口的缓存设计。

    # 接口返回格式

    1. 返回数据通用结构, 若接口异常code ！= 200, message异常信息不可为空
    {
  "code": "200",
  "message": "",
  "data": { }
}
    1. 项目中所有静态枚举统一通过一个接口返回
    2. 接口文档swagge、smartdoc，必须含有中文注释，返回字段以接口文档为准
        {
       "code": "200",
       "message": null,
       "data": {
           "memberLogStatus": [{
                   "value": 1,
                   "label": "激活"
               },
               {
                   "value": 2,
                   "label": "锁定"
               },
               {
                   "value": 3,
                   "label": "完善额度信息"
               },
               {
                   "value": 4,
                   "label": "修改额度信息"
               },
               {
                   "value": 5,
                   "label": "增信信息"
               }
           ],
           "memberLogStatusDescription": "会员日志类型枚举",
       }
   }
    1. 不得直接返回dto对象，不得直接接收dto对象。不管是入参、响应，必须封装一层vo供前端使用，只返回前端需要的字段、只接收前端需要传的字段
    2. 所有列表页面接口字段直接返回String形式,所有带状态或者枚举的字段例如status=1表示待审核，加上statusStr字段返回，例如时间直接返回相应的yyyy-MM-dd等字符串，不涉及到计算的总金额、总吨数、汇总等均以字符串形式返回 例如sum表示汇总，增加sumStr
    3. 有些操作按钮需根据状态判断是否显示，少于或者两个字段判断的前端来处理，如果该按钮需要两个以上字段来判断的话。后端封装成一个字段来供前端判断按钮是否展示 例如:showPaymentButton= 0
    4. 金额、重量后端需二次计算保证精度准确,以后端计算为准
    5. 后端所有接口必须自测，发布到测试环境接口文档中的所有接口默认都是自测过的
